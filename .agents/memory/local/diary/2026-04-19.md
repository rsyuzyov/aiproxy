# 2026-04-19

## Расследование висящих панелей cliproxy-api и 9router (контейнер 107)

**Симптомы:**
- `http://127.0.0.1:8317/management.html` — вечная загрузка, `/` работает
- `http://127.0.0.1:20128/dashboard` — вечная загрузка (307 → /login → висит 15s, 0 байт)

**Диагностика cliproxy-api (6.9.29):**
- `/management.html` существует локально (`/opt/cliproxy-api/static/management.html`, 2.3 МБ, валидный SPA HTML)
- Бинарь пишет 2.3 МБ в сокет (`Send-Q=2357070`), но клиент не получает HTTP-заголовков — curl висит 0 байт
- Не зависит от версии: воспроизводится на 6.9.26, 6.9.28, 6.9.29
- Не связано со свежим релизом (6.9.30 от 19.04 не про это)
- Бинарь держит 4 постоянных ESTAB-соединения на :443 (GitHub, Cloudflare, GCP)
- Остальные роуты (`/v0/management/status`, `/v1/models`, `/api/*`) отвечают быстро
- **Обход**: `disable-control-panel: true` в `config.yaml` → `/management.html` = 404 за 17 мс ✓

**Диагностика 9router (v0.3.96):**
- `/api/health` → 200 мгновенно; `/api/cloud/auth` → 405 мгновенно; `/api/auth/session`, `/login`, `/dashboard` — виснут 15s
- Strace: после `GET /login` next-server читает `/root/.9router/db.json` дважды, затем только eventfd wakeups (`read(fd=16, "\1\0\0\0...")`) — **deadlock в event-loop после чтения db.json**
- Не помогает: откат на v0.3.83, `--tray`, `--log`, bind на `127.0.0.1` (HOSTNAME=127.0.0.1), `requireLogin: false` в db.json
- Исходящих соединений next-server не делает (только FIN-WAIT-2/CLOSING от закрытых curl)
- В issues апстрима ([decolua/9router](https://github.com/decolua/9router)) ровно такого симптома нет

**Текущее состояние контейнера 107:**
- cliproxy-api: 6.9.29, `disable-control-panel: true`, API работает; `cliproxy-api-updater.timer` **disabled**
- 9router: v0.3.96, unit `--host 127.0.0.1 --port 20128 --tray --log`, `9router-updater.timer` **disabled**
- Конфиги забэкаплены: `config.yaml.bak.<ts>`, `db.json.bak`, `cli-proxy-api.broken-6.9.29`

**Дополнительное подтверждение (под конец дня):** скопировали `management.html` на рабочий комп, открыли через `file://` — панель работает и подключается к `http://aiproxy1.ag.local:8317`. То есть SPA, API, CORS и auth все ОК, баг **строго в HTTP-хендлере бинаря**, отдающем этот файл. Рабочий обход найден: открывать панель локально на рабочей машине.
