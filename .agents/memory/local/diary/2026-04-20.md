## Поставили ловушку (без рестарта xrdp)

Артефакт: [topics/xrdp.md](../topics/xrdp.md#ловля-причины-падения-chansrv-wrapper--dpkg-divert) (секция про wrapper)

- `/usr/local/sbin/xrdp-chansrv-wrapper.sh` — логирует START/EXIT с rc/signal в `/var/log/xrdp-chansrv-wrapper.log`, `ulimit -c unlimited`, cwd=`/var/crash/xrdp-chansrv/`
- `dpkg-divert --rename --divert /usr/sbin/xrdp-chansrv.real --add /usr/sbin/xrdp-chansrv`, `/usr/sbin/xrdp-chansrv` → symlink на wrapper
- `[ChansrvLogging]` в `/etc/xrdp/sesman.ini`: `LogFile=/var/log/xrdp-chansrv.${DISPLAY}.log`, `LogLevel=DEBUG`, `EnableSyslog=true`

## Оживили clipboard без logout — через ручной запуск chansrv

**Грабля:** xrdp-sesman **не перезапускает** chansrv при reconnect в persistent-сессию (видно в логах: последний `Calling exec chansrv` был 19:40, после reconnect 10:27 — только попытка connect, без exec). Это официальное by-design поведение 0.10.x.

**Рабочий трюк (проверено 2026-04-20):** запустить chansrv руками с правильным env, потом disconnect+reconnect RDP-клиентом:
```bash
systemd-run --unit=xrdp-chansrv-manual-10 --scope --quiet \
    --setenv=DISPLAY=:10 \
    --setenv=XAUTHORITY=/root/.Xauthority \
    --setenv=HOME=/root --setenv=USER=root --setenv=LANG=ru_RU.UTF-8 \
    --setenv=PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin \
    /usr/local/sbin/xrdp-chansrv-wrapper.sh </dev/null >/dev/null 2>&1 &
```
- chansrv создаёт `/run/xrdp/sockdir/0/xrdp_chansrv_socket_10`
- юзер делает disconnect mstsc → reconnect → xrdp-wm нового worker'а успешно коннектится к socket → clipboard восстановлен
- **logout не требуется**
