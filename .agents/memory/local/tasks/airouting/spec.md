# Спека: AI Routing

## Контекст

Нужно построить на `proxy1.ag.local` две параллельные, независимые системы маршрутизации:

- `sing-box` с входной точкой `:1080`
- `xray` с входной точкой `:8080`

Обе системы должны поддерживать одинаковый набор логических маршрутов, но не смешиваться между собой ни по конфигам, ни по сервисам, ни по механике переключения.

## Цели

Поддержать 5 логических маршрутов:

- `us-direct`
  `proxy1 -> us.prx.ontab.ru`
- `kz-us`
  `proxy1 -> kz -> us.prx.ontab.ru`
- `nl-us`
  `proxy1 -> nl -> us.prx.ontab.ru`
- `kz-direct`
  `proxy1 -> kz -> direct`
- `nl-direct`
  `proxy1 -> nl -> direct`

Дополнительно:

- для `sing-box` нужен ручной и автоматический выбор маршрута
- для `xray` нужен ручной выбор маршрута
- на `kz` и `nl` должен быть предусмотрен доступ для 5 пользователей
- все изменения должны быть обратимыми, с бэкапами конфигов

## Топология

Узлы:

- `proxy1.ag.local` — основной входной узел
- `kz.vps.ontab.ru` — промежуточный узел
- `nl.vps.ontab.ru` — промежуточный узел
- `us.prx.ontab.ru:8000` — внешний SOCKS5 upstream

## Принципы

- `sing-box` и `xray` работают параллельно и независимо
- входные точки не меняются:
  - `sing-box`: `1080`
  - `xray`: `8080`
- маршруты именуются одинаково в обоих стеках:
  - `us-direct`
  - `kz-us`
  - `nl-us`
  - `kz-direct`
  - `nl-direct`
- конфиги должны быть структурированы так, чтобы активный маршрут был очевиден и менялся в одном месте
- все промежуточные узлы должны быть готовы к масштабированию на 5 пользователей

## Функциональные требования

### 1. Sing-box на proxy1

`proxy1` должен предоставлять SOCKS5 вход на `:1080`.

Внутри `sing-box` должны быть описаны `outbound` для всех 5 маршрутов:

- `us-direct`
- `kz-us`
- `nl-us`
- `kz-direct`
- `nl-direct`

Поверх них должны быть реализованы два режима:

- ручной выбор маршрута через `selector`
- автоматический выбор через `urltest` только среди:
  - `us-direct`
  - `kz-us`
  - `nl-us`

Автоматический выбор не должен затрагивать:

- `kz-direct`
- `nl-direct`

Нужно предусмотреть один из двух вариантов работы:

Вариант A, предпочтительный:

- `:1080` — ручной selector
- дополнительный отдельный inbound, например `:1081` — авто-режим `urltest-us`

Вариант B:

- только `:1080`
- active outbound selector переключается через API/controller

Для простоты эксплуатации предпочтителен вариант A.

### 2. Xray на proxy1

`proxy1` должен предоставлять SOCKS5 вход на `:8080`.

Внутри одного `xray`-конфига должны быть перечислены все 5 `outbound`:

- `us-direct`
- `kz-us`
- `nl-us`
- `kz-direct`
- `nl-direct`

`routing` должен быть устроен так, чтобы весь входящий трафик `socks-in` направлялся в один активный `outboundTag`.

Активный маршрут должен задаваться в одном месте конфига.

Автоматический выбор fastest-route для `xray` на первом этапе не требуется.

Переключение маршрута для `xray` предполагается ручное:

- правка одного `outboundTag`
- проверка конфига
- рестарт сервиса

### 3. Промежуточные узлы kz и nl

На каждом узле должны параллельно существовать:

- `sing-box` endpoint для маршрутов `kz-*` / `nl-*`
- `xray` endpoint для маршрутов `kz-*` / `nl-*`

То есть оба узла должны уметь принимать соединения от `proxy1` и дальше:

- либо отправлять трафик в `us.prx.ontab.ru:8000`
- либо выпускать в `direct`

Для каждого узла нужны два логических режима выхода:

- `*-us`
- `*-direct`

### 4. Модель пользователей

Нужно обеспечить доступ для 5 пользователей к `kz` и `nl`.

Требование уточнить:

- это 5 пользователей для подключения с `proxy1` к промежуточным узлам
- или 5 внешних клиентов-людей, которые будут использовать `kz/nl` напрямую

На уровне спеки закладываем оба варианта совместимости.

Для `sing-box`:

- если протокол `hysteria2`, пользователи задаются в `inbound.users`

Для `xray`:

- если используется `vless`, пользователи задаются как отдельные `clients` с UUID
- если используется `socks`, пользователи задаются в `accounts`

Рекомендуемый принцип:

- 1 системная учётка для связи `proxy1 -> kz/nl`
- отдельный блок из 5 пользовательских учёток, если нужен прямой доступ людей к этим узлам

## Протокольная модель

### Sing-box

Рекомендуемая транспортная схема:

- `proxy1 -> kz/nl`: `hysteria2`
- `kz/nl -> us.prx`: `socks5`
- `kz/nl -> direct`: `direct`

Причина:

- хорошая скорость
- удобная встроенная механика `selector` и `urltest`
- уже показала лучшие результаты по latency, чем текущий `xray + reality`

### Xray

Рекомендуемая транспортная схема:

- `proxy1 -> kz/nl`: `vless + reality`
- `kz/nl -> us.prx`: `socks5`
- `kz/nl -> direct`: `freedom`

Причина:

- хорошая маскировка
- уже развернутая рабочая модель
- удобно держать как отдельный независимый стек

## Именование

Единое именование сущностей в обоих стеках.

Маршруты:

- `us-direct`
- `kz-us`
- `nl-us`
- `kz-direct`
- `nl-direct`

Рекомендуемые теги для входов:

- `socks-in`

Рекомендуемые теги для служебных групп в `sing-box`:

- `manual-selector`
- `auto-us`

## Конфигурационная модель

### Sing-box

Один конфиг на каждом хосте.

На `proxy1`:

- 1 inbound `:1080`
- опционально 1 inbound `:1081`
- набор outbound
- selector
- urltest
- route rules, направляющие нужный inbound в нужную группу

На `kz` и `nl`:

- inbound для `hy2`
- outbound:
  - `us-socks`
  - `direct`
- при необходимости route/final

### Xray

Один конфиг на каждом хосте.

На `proxy1`:

- 1 inbound `:8080`
- 5 outbound
- routing rule с одним активным `outboundTag`

На `kz` и `nl`:

- inbound для `vless+reality` или другого выбранного xray-протокола
- outbound:
  - `us-socks`
  - `direct`

## Управление и переключение

### Sing-box

Должно быть 2 режима:

Ручной:

- через `selector`
- желательно через API/controller или отдельный inbound

Автоматический:

- через `urltest`
- только для маршрутов до `us.prx`

Параметры `urltest` надо задать явно:

- URL проверки
- интервал проверки
- tolerance
- timeout

Рекомендуемая проверка:

- HTTPS URL небольшого размера
- стабильно доступный
- не завязанный на блокируемые домены

### Xray

Ручное переключение.

Требования:

- активный маршрут меняется в одном поле
- после изменения обязательно:
  - `xray -test`
  - `systemctl restart`
  - `systemctl is-active`

## Наблюдаемость

Для обоих стеков нужны базовые средства проверки:

- проверка активного маршрута
- проверка внешнего IP
- проверка времени ответа
- проверка статуса сервисов
- проверка listening ports

Минимальные контрольные операции:

- запрос к `https://api.ipify.org`
- запрос к `https://ifconfig.me`
- замер `curl -w`
- `systemctl is-active`
- `ss -ltnp`

## Безопасность

- все конфиги перед заменой бэкапятся
- промежуточные узлы не должны оставлять лишние публичные порты открытыми
- аутентификация на `kz` и `nl` обязательна
- для user-based inbound нужно закладывать отдельные учётки
- секреты не должны дублироваться без необходимости между разными ролями

Отдельно:

- учётки для `proxy1 -> kz/nl`
- учётки для пользователей, если нужен прямой клиентский доступ

## Ограничения

- `proxy1` входы фиксированы:
  - `1080` — sing-box
  - `8080` — xray
- `sing-box` и `xray` не смешиваются
- автоматика latency-based выбора делается только в `sing-box`
- `xray` на первом этапе без автоматического выбора

## Этапы реализации

### Этап 1. Нормализация имен и маршрутов

- зафиксировать имена маршрутов
- обновить конфиги с использованием `us.prx.ontab.ru`

### Этап 2. Sing-box

- `proxy1`:
  - manual selector
  - auto urltest
  - маршруты `us-direct`, `kz-us`, `nl-us`, `kz-direct`, `nl-direct`
- `kz`:
  - `hy2` inbound
  - outbound `us-socks` и `direct`
- `nl`:
  - `hy2` inbound
  - outbound `us-socks` и `direct`

### Этап 3. Xray

- `proxy1`:
  - единый конфиг с 5 outbound
  - один активный `outboundTag`
- `kz`:
  - xray inbound
  - outbound `us-socks` и `direct`
- `nl`:
  - xray inbound
  - outbound `us-socks` и `direct`

### Этап 4. Пользователи

- добавить модель 5 пользователей на `kz` и `nl`
- развести системные и пользовательские учётки

### Этап 5. Проверки

- внешний IP
- latency по каждому маршруту
- ручное переключение
- auto-switch в `sing-box`
- rollback по бэкапам

## Критерии готовности

Считаем задачу выполненной, когда:

- `proxy1:1080` работает через `sing-box`
- `proxy1:8080` работает через `xray`
- доступны все 5 логических маршрутов в обоих стеках
- `sing-box` умеет manual + auto выбор
- `xray` умеет ручной выбор без перестройки нескольких конфигов
- `kz` и `nl` умеют маршруты `-> us.prx` и `-> direct`
- предусмотрена схема добавления 5 пользователей
- есть бэкапы и понятный rollback

## Открытые вопросы

Их надо закрыть перед реализацией:

1. Нужен ли отдельный auto-port для `sing-box`, например `1081`, или авто-режим будет переключаться только через API?
2. 5 пользователей на `kz` и `nl` — это пользователи `proxy1` или внешние клиенты-люди?
3. Для `xray` на `kz/nl` точно оставляем `vless+reality`, или допустим другой транспорт ради упрощения?
4. На `nl` уже есть готовый `sing-box` endpoint или его поднимать с нуля?
5. Нужны ли отдельные учётки для `kz` и `nl`, или можно единый набор пользователей на оба узла?
